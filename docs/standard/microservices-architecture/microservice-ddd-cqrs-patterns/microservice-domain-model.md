# <a name="designing-a-microservice-domain-model"></a><span data-ttu-id="9713c-103">Conception d’un modèle de domaine de microservice</span><span class="sxs-lookup"><span data-stu-id="9713c-103">Designing a microservice domain model</span></span>

<span data-ttu-id="9713c-104">*Définir un seul modèle de domaine complet pour chaque microservice métier ou contexte limité*</span><span class="sxs-lookup"><span data-stu-id="9713c-104">*Define one rich domain model for each business microservice or Bounded Context*</span></span>

<span data-ttu-id="9713c-105">Votre objectif consiste à créer un seul modèle de domaine cohésif pour chaque microservice métier ou contexte limité.</span><span class="sxs-lookup"><span data-stu-id="9713c-105">Your goal is to create a single cohesive domain model for each business microservice or Bounded Context (BC).</span></span> <span data-ttu-id="9713c-106">N’oubliez pas, toutefois, qu’un contexte limité ou microservice métier peuvent parfois se composer de plusieurs services physiques qui partagent un modèle de domaine unique.</span><span class="sxs-lookup"><span data-stu-id="9713c-106">Keep in mind, however, that a BC or business microservice could sometimes be composed of several physical services that share a single domain model.</span></span> <span data-ttu-id="9713c-107">Le modèle de domaine doit capturer les règles, le comportement, le langage informatique de l’entreprise et les contraintes du contexte limité ou microservice métier unique qu’il représente.</span><span class="sxs-lookup"><span data-stu-id="9713c-107">The domain model must capture the rules, behavior, business language, and constraints of the single Bounded Context or business microservice that it represents.</span></span>

## <a name="the-domain-entity-pattern"></a><span data-ttu-id="9713c-108">Le modèle Entité de domaine</span><span class="sxs-lookup"><span data-stu-id="9713c-108">The Domain Entity pattern</span></span>

<span data-ttu-id="9713c-109">Les entités représentent des objets de domaine. Elles sont principalement définies par leur identité, leur continuité et leur persistance dans le temps et non uniquement par les attributs qui les composent.</span><span class="sxs-lookup"><span data-stu-id="9713c-109">Entities represent domain objects and are primarily defined by their identity, continuity, and persistence over time, and not only by the attributes that comprise them.</span></span> <span data-ttu-id="9713c-110">Comme le décrit Eric Evans, « un objet principalement défini par son identité est appelé entité ».</span><span class="sxs-lookup"><span data-stu-id="9713c-110">As Eric Evans says, “an object primarily defined by its identity is called an Entity.”</span></span> <span data-ttu-id="9713c-111">Les entités sont très importantes dans le modèle de domaine, puisqu’elles constituent la base d’un modèle.</span><span class="sxs-lookup"><span data-stu-id="9713c-111">Entities are very important in the domain model, since they are the base for a model.</span></span> <span data-ttu-id="9713c-112">Par conséquent, vous devez les identifier et les concevoir soigneusement.</span><span class="sxs-lookup"><span data-stu-id="9713c-112">Therefore, you should identify and design them carefully.</span></span>

<span data-ttu-id="9713c-113">*L’identité d’une entité peut traverser plusieurs microservices ou contextes limités.*</span><span class="sxs-lookup"><span data-stu-id="9713c-113">*An entity’s identity can cross multiple microservices or Bounded Contexts.*</span></span>

<span data-ttu-id="9713c-114">La même identité (mais pas la même entité) peut être modélisée dans plusieurs contextes limités ou microservices.</span><span class="sxs-lookup"><span data-stu-id="9713c-114">The same identity (though not the same entity) can be modeled across multiple Bounded Contexts or microservices.</span></span> <span data-ttu-id="9713c-115">Toutefois, cela n’implique pas que la même entité, avec les mêmes attributs et la même logique, peut être implémentée dans plusieurs contextes limités.</span><span class="sxs-lookup"><span data-stu-id="9713c-115">However, that does not imply that the same entity, with the same attributes and logic would be implemented in multiple Bounded Contexts.</span></span> <span data-ttu-id="9713c-116">En effet, les entités incluses dans chaque contexte limité limitent leurs attributs et comportements à ceux nécessaires dans le domaine du contexte limité.</span><span class="sxs-lookup"><span data-stu-id="9713c-116">Instead, entities in each Bounded Context limit their attributes and behaviors to those required in that Bounded Context’s domain.</span></span>

<span data-ttu-id="9713c-117">Par exemple, l’entité Buyer peut avoir la plupart des attributs d’une personne définis dans l’entité User dans le profil ou microservice d’identité, notamment son identité.</span><span class="sxs-lookup"><span data-stu-id="9713c-117">For instance, the buyer entity might have most of a person’s attributes that are defined in the user entity in the profile or identity microservice, including the identity.</span></span> <span data-ttu-id="9713c-118">Mais l’entité Buyer dans le microservice de passation de commandes peut avoir moins d’attributs, car seules certaines données sur l’acheteur sont liées au processus de commande.</span><span class="sxs-lookup"><span data-stu-id="9713c-118">But the buyer entity in the ordering microservice might have fewer attributes, because only certain buyer data is related to the order process.</span></span> <span data-ttu-id="9713c-119">Le contexte de chaque microservice ou contexte limité exerce un impact sur son modèle de domaine.</span><span class="sxs-lookup"><span data-stu-id="9713c-119">The context of each microservice or Bounded Context impacts its domain model.</span></span>

<span data-ttu-id="9713c-120">*Les entités de domaine doivent implémenter le comportement en plus des attributs de données.*</span><span class="sxs-lookup"><span data-stu-id="9713c-120">*Domain entities must implement behavior in addition to implementing data attributes*</span></span>

<span data-ttu-id="9713c-121">Une entité de domaine en conception pilotée par le domaine (Domain Driver Design, DDD) doit implémenter la logique de domaine ou le comportement lié aux données de l’entité (objet auquel vous accédez dans la mémoire).</span><span class="sxs-lookup"><span data-stu-id="9713c-121">A domain entity in DDD must implement the domain logic or behavior related to the entity data (the object accessed in memory).</span></span> <span data-ttu-id="9713c-122">Par exemple, dans le cadre d’une classe d’entité de commande, vous devez implémenter la logique et les opérations métier en tant que méthodes pour des tâches comme l’ajout d’un article, la validation des données et le calcul du total.</span><span class="sxs-lookup"><span data-stu-id="9713c-122">For example, as part of an order entity class you must have business logic and operations implemented as methods for tasks such as adding an order item, data validation, and total calculation.</span></span> <span data-ttu-id="9713c-123">Les méthodes de l’entité s’occupent des invariants et des règles de l’entité au lieu de répartir ces règles dans toute la couche Application.</span><span class="sxs-lookup"><span data-stu-id="9713c-123">The entity’s methods take care of the invariants and rules of the entity instead of having those rules spread across the application layer.</span></span>

<span data-ttu-id="9713c-124">La figure 9-8 illustre une entité de domaine qui implémente non seulement des attributs de données, mais aussi des opérations ou méthodes avec une logique de domaine associée.</span><span class="sxs-lookup"><span data-stu-id="9713c-124">Figure 9-8 shows a domain entity that implements not only data attributes but operations or methods with related domain logic.</span></span>

<span data-ttu-id="9713c-127">Bien entendu, vous pouvez parfois avoir des entités qui n’implémentent pas de logique dans le cadre de la classe d’entité.</span><span class="sxs-lookup"><span data-stu-id="9713c-127">Of course, sometimes you can have entities that do not implement any logic as part of the entity class.</span></span> <span data-ttu-id="9713c-128">Cela peut se produire dans les entités enfants au sein d’un agrégat si l’entité enfant n’a pas de logique spéciale, car la plupart de la logique est définie dans la racine d’agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-128">This can happen in child entities within an aggregate if the child entity does not have any special logic because most of the logic is defined in the aggregate root.</span></span> <span data-ttu-id="9713c-129">Si vous avez un microservice complexe qui contient beaucoup de logique implémentée dans les classes de service au lieu des entités de domaine, vous risquez de vous retrouver avec le modèle de domaine anémique, décrit dans la section suivante.</span><span class="sxs-lookup"><span data-stu-id="9713c-129">If you have a complex microservice that has a lot of logic implemented in the service classes instead of in the domain entities, you could be falling into the anemic domain model, explained in the following section.</span></span>

### <a name="rich-domain-model-versus-anemic-domain-model"></a><span data-ttu-id="9713c-130">Modèle de domaine riche et modèle de domaine anémique</span><span class="sxs-lookup"><span data-stu-id="9713c-130">Rich domain model versus anemic domain model</span></span>

<span data-ttu-id="9713c-131">Dans son billet [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler décrit un modèle de domaine anémique de la façon suivante :</span><span class="sxs-lookup"><span data-stu-id="9713c-131">In his post [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler describes an anemic domain model this way:</span></span>

<span data-ttu-id="9713c-132">Le symptôme de base d’un modèle de domaine anémique est qu’à première vue, il ressemble à la réalité.</span><span class="sxs-lookup"><span data-stu-id="9713c-132">The basic symptom of an Anemic Domain Model is that at first blush it looks like the real thing.</span></span> <span data-ttu-id="9713c-133">Il s’agit d’objets souvent nommés d’après les noms figurant dans l’espace de domaine. Ces objets sont connectés avec les riches relations et structure que présentent les véritables modèles de domaine.</span><span class="sxs-lookup"><span data-stu-id="9713c-133">There are objects, many named after the nouns in the domain space, and these objects are connected with the rich relationships and structure that true domain models have.</span></span> <span data-ttu-id="9713c-134">L’imposture apparaît quand vous examinez le comportement et que vous réalisez qu’il n’y a pas vraiment de comportement sur ces objets, ce qui en fait ni plus ni moins des contenants de méthodes getter et setter.</span><span class="sxs-lookup"><span data-stu-id="9713c-134">The catch comes when you look at the behavior, and you realize that there is hardly any behavior on these objects, making them little more than bags of getters and setters.</span></span>

<span data-ttu-id="9713c-135">Bien entendu, quand vous utilisez un modèle de domaine anémique, il est utilisé à partir d’un ensemble d’objets de service (généralement nommé *couche métier*) qui capture toute la logique métier ou du domaine.</span><span class="sxs-lookup"><span data-stu-id="9713c-135">Of course, when you use an anemic domain model, those data models will be used from a set of service objects (traditionally named the *business layer*) which capture all the domain or business logic.</span></span> <span data-ttu-id="9713c-136">La couche métier est posée sur le modèle de données et utilise ce dernier simplement comme des données.</span><span class="sxs-lookup"><span data-stu-id="9713c-136">The business layer sits on top of the data model and uses the data model just as data.</span></span>

<span data-ttu-id="9713c-137">Le modèle de domaine anémique est simplement une conception de style procédural.</span><span class="sxs-lookup"><span data-stu-id="9713c-137">The anemic domain model is just a procedural style design.</span></span> <span data-ttu-id="9713c-138">Les objets d’entité anémique ne sont pas des objets réels, car ils n’ont pas de comportement (méthodes).</span><span class="sxs-lookup"><span data-stu-id="9713c-138">Anemic entity objects are not real objects because they lack behavior (methods).</span></span> <span data-ttu-id="9713c-139">Ils contiennent uniquement des propriétés de données et ne correspondent donc pas à une conception orientée objet.</span><span class="sxs-lookup"><span data-stu-id="9713c-139">They only hold data properties and thus it is not object-oriented design.</span></span> <span data-ttu-id="9713c-140">En plaçant tout le comportement dans des objets de service (la couche métier), vous finissez par obtenir du [code spaghetti](https://en.wikipedia.org/wiki/Spaghetti_code) ou des [scripts de transaction](https://martinfowler.com/eaaCatalog/transactionScript.html). Vous perdez donc les avantages liés au modèle de domaine.</span><span class="sxs-lookup"><span data-stu-id="9713c-140">By putting all the behavior out into service objects (the business layer) you essentially end up with [spaghetti code](https://en.wikipedia.org/wiki/Spaghetti_code) or [transaction scripts](https://martinfowler.com/eaaCatalog/transactionScript.html), and therefore you lose the advantages that a domain model provides.</span></span>

<span data-ttu-id="9713c-141">Quand même, si votre microservice ou contexte limité est très simple (un service CRUD), le modèle de domaine anémique sous la forme d’objets d’entité avec seulement des propriétés de données peut vous suffire, et l’implémentation de modèles DDD plus complexes peut ne pas valoir le coup.</span><span class="sxs-lookup"><span data-stu-id="9713c-141">Regardless, if your microservice or Bounded Context is very simple (a CRUD service), the anemic domain model in the form of entity objects with just data properties might be good enough, and it might not be worth implementing more complex DDD patterns.</span></span> <span data-ttu-id="9713c-142">Dans ce cas, il s’agit simplement d’un modèle de persistance, car vous avez intentionnellement créé une entité avec des données uniquement pour les besoins de CRUD.</span><span class="sxs-lookup"><span data-stu-id="9713c-142">In that case, it will be simply a persistence model, because you have intentionally created an entity with only data for CRUD purposes.</span></span>

<span data-ttu-id="9713c-143">C’est pourquoi les architectures des microservices conviennent parfaitement à une approche à plusieurs architectures dépendant de chaque contexte limité.</span><span class="sxs-lookup"><span data-stu-id="9713c-143">That is why microservices architectures are perfect for a multi-architectural approach depending on each Bounded Context.</span></span> <span data-ttu-id="9713c-144">Par exemple, dans eShopOnContainers, le microservice de passation de commandes implémente des modèles DDD, contrairement au microservice de catalogue, qui est un simple service CRUD.</span><span class="sxs-lookup"><span data-stu-id="9713c-144">For instance, in eShopOnContainers, the ordering microservice implements DDD patterns, but the catalog microservice, which is a simple CRUD service, does not.</span></span>

<span data-ttu-id="9713c-145">Certaines personnes disent que le modèle de domaine anémique est un anti-modèle.</span><span class="sxs-lookup"><span data-stu-id="9713c-145">Some people say that the anemic domain model is an anti-pattern.</span></span> <span data-ttu-id="9713c-146">Il dépend vraiment de ce que vous implémentez.</span><span class="sxs-lookup"><span data-stu-id="9713c-146">It really depends on what you are implementing.</span></span> <span data-ttu-id="9713c-147">Si le microservice que vous créez est suffisamment simple (par exemple, un service CRUD), le modèle de domaine anémique n’est pas un anti-modèle.</span><span class="sxs-lookup"><span data-stu-id="9713c-147">If the microservice you are creating is simple enough (for example, a CRUD service), following the anemic domain model it is not an anti-pattern.</span></span> <span data-ttu-id="9713c-148">Toutefois, si vous avez besoin de vous confronter à la complexité du domaine d’un microservice dont le grand nombre de règles d’entreprise est en constante évolution, le modèle de domaine anémique peut être un anti-modèle pour ce microservice ou contexte limité.</span><span class="sxs-lookup"><span data-stu-id="9713c-148">However, if you need to tackle the complexity of a microservice’s domain that has a lot of ever-changing business rules, the anemic domain model might be an anti-pattern for that microservice or Bounded Context.</span></span> <span data-ttu-id="9713c-149">Dans ce cas, une conception en tant que modèle riche avec des entités contenant des données en plus d’un comportement, ainsi que l’implémentation de modèles DDD supplémentaires (agrégats, objets de valeur, etc.) peut offrir des avantages énormes pour le succès à long terme d’un tel microservice.</span><span class="sxs-lookup"><span data-stu-id="9713c-149">In that case, designing it as a rich model with entities containing data plus behavior as well as implementing additional DDD patterns (aggregates, value objects, etc.) might have huge benefits for the long-term success of such a microservice.</span></span>

### <a name="the-value-object-pattern"></a><span data-ttu-id="9713c-154">Le modèle Objet de valeur</span><span class="sxs-lookup"><span data-stu-id="9713c-154">The Value Object pattern</span></span>

<span data-ttu-id="9713c-155">Comme l’a noté Eric Evans, « de nombreux objets n’ont pas d’identité conceptuelle.</span><span class="sxs-lookup"><span data-stu-id="9713c-155">As Eric Evans has noted, “Many objects do not have conceptual identity.</span></span> <span data-ttu-id="9713c-156">Ces objets décrivent certaines caractéristiques d’une chose ».</span><span class="sxs-lookup"><span data-stu-id="9713c-156">These objects describe certain characteristics of a thing.”</span></span>

<span data-ttu-id="9713c-157">Une entité nécessite une identité, mais il existe de nombreux objets dans un système qui n’en ont pas besoin, comme le modèle Objet de valeur.</span><span class="sxs-lookup"><span data-stu-id="9713c-157">An entity requires an identity, but there are many objects in a system that do not, like the Value Object pattern.</span></span> <span data-ttu-id="9713c-158">Un objet de valeur est un objet sans identité conceptuelle qui décrit un aspect de domaine.</span><span class="sxs-lookup"><span data-stu-id="9713c-158">A value object is an object with no conceptual identity that describes a domain aspect.</span></span> <span data-ttu-id="9713c-159">Ce sont des objets que vous instanciez pour représenter des éléments de conception qui ne vous concernent que temporairement.</span><span class="sxs-lookup"><span data-stu-id="9713c-159">These are objects that you instantiate to represent design elements that only concern you temporarily.</span></span> <span data-ttu-id="9713c-160">Il vous importe de savoir *ce* qu’ils sont, pas *qui* ils sont.</span><span class="sxs-lookup"><span data-stu-id="9713c-160">You care about *what* they are, not *who* they are.</span></span> <span data-ttu-id="9713c-161">Les nombres et les chaînes en sont des exemples, mais il peut aussi s’agir de concepts plus généraux comme des groupes d’attributs.</span><span class="sxs-lookup"><span data-stu-id="9713c-161">Examples include numbers and strings, but can also be higher-level concepts like groups of attributes.</span></span>

<span data-ttu-id="9713c-162">Une entité dans un microservice peut ne pas être une entité dans un autre microservice, puisque dans le deuxième cas, le contexte limité peut avoir une signification différente.</span><span class="sxs-lookup"><span data-stu-id="9713c-162">Something that is an entity in a microservice might not be an entity in another microservice, because in the second case, the Bounded Context might have a different meaning.</span></span> <span data-ttu-id="9713c-163">Par exemple, une adresse dans une application de commerce électronique peut ne pas avoir du tout d’identité, puisqu’elle peut représenter uniquement un groupe d’attributs du profil de client d’une personne ou d’une entreprise.</span><span class="sxs-lookup"><span data-stu-id="9713c-163">For example, an address in an e-commerce application might not have an identity at all, since it might only represent a group of attributes of the customer’s profile for a person or company.</span></span> <span data-ttu-id="9713c-164">Dans ce cas, l’adresse doit être classée en tant qu’objet de valeur.</span><span class="sxs-lookup"><span data-stu-id="9713c-164">In this case, the address should be classified as a value object.</span></span> <span data-ttu-id="9713c-165">Toutefois, dans une application destinée à un fournisseur d’énergie électrique, l’adresse du client peut s’avérer importante pour le domaine d’entreprise.</span><span class="sxs-lookup"><span data-stu-id="9713c-165">However, in an application for an electric power utility company, the customer address could be important for the business domain.</span></span> <span data-ttu-id="9713c-166">Par conséquent, l’adresse doit avoir une identité pour que le système de facturation puisse être directement lié à l’adresse.</span><span class="sxs-lookup"><span data-stu-id="9713c-166">Therefore, the address must have an identity so the billing system can be directly linked to the address.</span></span> <span data-ttu-id="9713c-167">Dans cet exemple, une adresse doit être classée en tant qu’entité de domaine.</span><span class="sxs-lookup"><span data-stu-id="9713c-167">In that case, an address should be classified as a domain entity.</span></span>

<span data-ttu-id="9713c-168">Une personne désignée par un prénom et un nom est généralement une entité, car elle a une identité, même si ce prénom et ce nom coïncident avec un autre ensemble de valeurs, par exemple s’ils font également référence à une autre personne.</span><span class="sxs-lookup"><span data-stu-id="9713c-168">A person with a name and surname is usually an entity because a person has identity, even if the name and surname coincide with another set of values, such as if those names also refers to a different person.</span></span>

<span data-ttu-id="9713c-169">Les objets de valeur sont difficiles à gérer dans les bases de données relationnelles et les ORM comme EF, alors que dans les bases de données orientées document, ils sont plus faciles à implémenter et à utiliser.</span><span class="sxs-lookup"><span data-stu-id="9713c-169">Value objects are hard to manage in relational databases and ORMs like EF, whereas in document oriented databases they are easier to implement and use.</span></span>

### <a name="the-aggregate-pattern"></a><span data-ttu-id="9713c-176">Le modèle Agrégat</span><span class="sxs-lookup"><span data-stu-id="9713c-176">The Aggregate pattern</span></span>

<span data-ttu-id="9713c-177">Un modèle de domaine contient des clusters d’entités de données et de processus différents qui peuvent contrôler une importante part des fonctionnalités, comme le déroulement des commandes ou l’inventaire.</span><span class="sxs-lookup"><span data-stu-id="9713c-177">A domain model contains clusters of different data entities and processes that can control a significant area of functionality, such as order fulfilment or inventory.</span></span> <span data-ttu-id="9713c-178">Un agrégat est une unité DDD plus fine qui décrit un cluster ou groupe d’entités et de comportements pouvant être traité comme une unité cohésive.</span><span class="sxs-lookup"><span data-stu-id="9713c-178">A more fine-grained DDD unit is the aggregate, which describes a cluster or group of entities and behaviors that can be treated as a cohesive unit.</span></span>

<span data-ttu-id="9713c-179">En règle générale, vous définissez un agrégat selon les transactions dont vous avez besoin.</span><span class="sxs-lookup"><span data-stu-id="9713c-179">You usually define an aggregate based on the transactions that you need.</span></span> <span data-ttu-id="9713c-180">Une commande qui contient aussi une liste d’articles constitue un exemple classique.</span><span class="sxs-lookup"><span data-stu-id="9713c-180">A classic example is an order that also contains a list of order items.</span></span> <span data-ttu-id="9713c-181">Un article correspond généralement à une entité.</span><span class="sxs-lookup"><span data-stu-id="9713c-181">An order item will usually be an entity.</span></span> <span data-ttu-id="9713c-182">Mais il s’agit d’une entité enfant au sein de l’agrégat de commande, qui contient aussi l’entité de commande en tant qu’entité racine, généralement appelée racine d’agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-182">But it will be a child entity within the order aggregate, which will also contain the order entity as its root entity, typically called an aggregate root.</span></span>

<span data-ttu-id="9713c-183">Il peut s’avérer difficile d’identifier les agrégats.</span><span class="sxs-lookup"><span data-stu-id="9713c-183">Identifying aggregates can be hard.</span></span> <span data-ttu-id="9713c-184">Un agrégat est un groupe d’objets qui doivent être cohérents ensemble, mais vous ne pouvez pas simplement choisir un groupe d’objets et les appeler agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-184">An aggregate is a group of objects that must be consistent together, but you cannot just pick a group of objects and label them an aggregate.</span></span> <span data-ttu-id="9713c-185">Vous devez commencer avec un concept de domaine et réfléchir aux entités utilisées dans les transactions les plus courantes liées à ce concept.</span><span class="sxs-lookup"><span data-stu-id="9713c-185">You must start with a domain concept and think about the entities that are used in the most common transactions related to that concept.</span></span> <span data-ttu-id="9713c-186">Ce sont ces entités devant être cohérentes du point de vue transactionnel qui forment un agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-186">Those entities that need to be transactionally consistent are what forms an aggregate.</span></span> <span data-ttu-id="9713c-187">Réfléchir aux opérations transactionnelles constitue probablement la meilleure façon d’identifier des agrégats.</span><span class="sxs-lookup"><span data-stu-id="9713c-187">Thinking about transaction operations is probably the best way to identify aggregates.</span></span>

### <a name="the-aggregate-root-or-root-entity-pattern"></a><span data-ttu-id="9713c-188">Le modèle Racine d’agrégat ou Entité racine</span><span class="sxs-lookup"><span data-stu-id="9713c-188">The Aggregate Root or Root Entity pattern</span></span>

<span data-ttu-id="9713c-189">Un agrégat se compose d’au moins une entité : la racine d’agrégat, également appelée entité racine ou entité principale.</span><span class="sxs-lookup"><span data-stu-id="9713c-189">An aggregate is composed of at least one entity: the aggregate root, also called root entity or primary entity.</span></span> <span data-ttu-id="9713c-190">De plus, il peut avoir plusieurs entités enfants et objets de valeur, qui fonctionnent tous ensemble pour implémenter le comportement et les transactions nécessaires.</span><span class="sxs-lookup"><span data-stu-id="9713c-190">Additionally, it can have multiple child entities and value objects, with all entities and objects working together to implement required behavior and transactions.</span></span>

<span data-ttu-id="9713c-191">Une racine d’agrégat a pour but de garantir la cohérence de l’agrégat ; elle doit être le seul point d’entrée pour les mises à jour de l’agrégat par le biais de méthodes ou d’opérations dans la classe de racine d’agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-191">The purpose of an aggregate root is to ensure the consistency of the aggregate; it should be the only entry point for updates to the aggregate through methods or operations in the aggregate root class.</span></span> <span data-ttu-id="9713c-192">Vous devez apporter des modifications aux entités au sein de l’agrégat uniquement par le biais de la racine d’agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-192">You should make changes to entities within the aggregate only via the aggregate root.</span></span> <span data-ttu-id="9713c-193">Celle-ci protège la cohérence de l’agrégat, en tenant compte de tous les invariants et de toutes les règles de cohérence à respecter dans votre agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-193">It is the aggregate’s consistency guardian, taking into account all the invariants and consistency rules you might need to comply with in your aggregate.</span></span> <span data-ttu-id="9713c-194">Si vous modifiez séparément une entité enfant ou un objet de valeur, la racine d’agrégat ne peut pas vérifier que l’agrégat est dans un état valide.</span><span class="sxs-lookup"><span data-stu-id="9713c-194">If you change a child entity or value object independently, the aggregate root cannot ensure that the aggregate is in a valid state.</span></span> <span data-ttu-id="9713c-195">Il serait alors comme une table avec un pied qui bouge.</span><span class="sxs-lookup"><span data-stu-id="9713c-195">It would be like a table with a loose leg.</span></span> <span data-ttu-id="9713c-196">Le maintien de la cohérence est le principal but de la racine d’agrégat.</span><span class="sxs-lookup"><span data-stu-id="9713c-196">Maintaining consistency is the main purpose of the aggregate root.</span></span>

<span data-ttu-id="9713c-197">Dans la figure 9-9, vous pouvez voir des exemples d’agrégats comme l’agrégat d’acheteur (Buyer), qui contient une seule entité (la racine d’agrégat Buyer).</span><span class="sxs-lookup"><span data-stu-id="9713c-197">In Figure 9-9, you can see sample aggregates like the buyer aggregate, which contains a single entity (the aggregate root Buyer).</span></span> <span data-ttu-id="9713c-198">L’agrégat de commande (Order) contient plusieurs entités et un objet de valeur.</span><span class="sxs-lookup"><span data-stu-id="9713c-198">The order aggregate contains multiple entities and a value object.</span></span>
